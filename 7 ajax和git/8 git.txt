没有版本控制软件 文件的版本控制会有4大问题：操作麻烦（复制粘贴重命名）、命名不规范（通过文件名无法知道做了什么修改）、容易丢失、协作困难（需要手动合并每个人对项目文件的修改）。

版本控制软件好处：操作简便、易于对比和回溯、不易丢失、协作方便

版本控制系统三类：本地版本控制系统、集中化版本控制系统、分布式版本控制系统。
git是开源分布式版本控制系统，是目前世界上最先进最流行的版本控制系统。

svn是记录文件差异：好处节约磁盘空间、坏处耗时获取每个文件版本都是从原文件加上各种差异化得到。
git是记录快照 类似直接备份 如果文件没有改动 就建立一个链接 指向旧版本文件。好处版本切换非常快。
git近乎所有操作都是本地执行。


git中的三个区域：工作区、暂存区、git仓库。
git中的三种状态：已修改、已暂存、已提交。分别对应三种区域。
git工作流程：在工作区中修改文件，将需要下次提交的更改进行暂存，提交更新把暂存区文件快照永久性存储到git仓库。

git基本操作
1 获取git仓库的两种方式
    将尚未进行版本控制的本地目录转换为git仓库  
    从其他服务器克隆一个已存在的git仓库

2 现有目录中初始化仓库
    项目目录打开命令行 输入git init  // 项目里可以有文件
    执行后会创建一个.git隐藏目录，这个目录就是git仓库，这里包含了初始的必要文件，是git仓库的组成部分。


3 工作区文件中的四种状态
    未跟踪：未被git管理的文件
    未修改：工作区中文件的内容和git仓库中文件的内容一致
    已修改：工作区中文件的内容和git仓库内容不一致
    已暂存：放入暂存区了
    git操作的终极结果是让工作区中的文件都处于未修改状态。

4 查看文件的状态
    git status
    git status -s //以精简的方式显示文件的状态

5 跟踪新文件
    git add 文件名  //执行后文件从untracked files下变到Changes to be committed下面，此时文件被跟踪并处于暂存状态； 精简显示：A（未跟踪时是??）
    git add . //为当前目录

6 提交更新
    git commit -m "描述" //提交后 git stauts显示nothing to commit，working tree clean；精简显示：无


7 对已提交的文件进行修改
    修改后git status显示文件在Changes not staged for commit下；说明文件已修改 还没放到暂存区。精简 显示为红M


8 暂存已修改的文件
    再次运行git add 文件名 //Changes to be committed 精简为绿M
    git add有三个功能：可以开始跟踪新文件放到暂存区；把已跟踪，且已修改的文件放到暂存区；把有冲突的文件标记为已解决状态

9 提交已暂存文件
    再次运行git commit -m "描述" 文件名

10 撤销文件的修改
    git checkout -- 文件名 //撤销文件的修改 保存为和仓库一致。  本质就是用git参考的文件覆盖工作区中文件。

11 向暂存区一次性添加多个文件
    git add .

12 取消文件的暂存
    git reset HEAD 文件名   // 多个用. 

13 跳过使用暂存区直接提交
    git commit -a -m "描述"  // -a 选项会直接把所有已经跟踪过的文件 提交到仓库 跳过暂存区

14 移除文件
    git rm -f 文件名 //从仓库和工作区同时移除对应的文件 精简 显示为绿D
    git rm --cached 文件名 //从仓库移除指定文件 但是保留工作区中对应的文件  精简 显示为绿D  下次提交时候 就没有了

15 忽略文件
    有些文件 我们不要被git管理 也不想总显示为未跟踪状态，此时创建一个.gitignore的配置文件，列出要忽略文件的匹配模式即可。
    写法见.gitignore的配置文件注释

16 查看提交历史
    git log  //最新提交在上面
    git log -2 //只展示最新的两条提交历史
    git log -2 --pretty=oneline //在一行上展示 最近两条提交历史
    git log -2 --pretty=format:"%h | %an | %ar | %s" //在一行上展示最近两条提交的历史信息，并自定义输出的格式；%h简写哈希值 %an作者 %ar日期 %s提交描述

17 回退到指定的版本
    git log --pretty=oneline  //查看版本hash
    git reset --hard <CommitID> //根据hash id 回退到指定的版本

    git reflog --pretty=oneline //假如你已经在旧版本中了 使用此命令 可以看见全部的版本
    git reset --hard <CommitID> //再次根据hash id 回退到指定的版本

5 ssh key
    SSH key 的作用:实现本地仓库和 Github 之间免登录的加密数据传输,
    SSH key 的好处:免登录身份认证、数据加密传输。
    SSH key 由两部分组成，分别是:
    ① id_rsa(私钥文件，存放于客户端的电脑中即可)
    ② id_rsa.pub(公钥文件，需要配置到 Github 中)

6 生成ssh key
    ① 打开 Git Bash
    ② 粘贴如下的命令，并将 your _email@example.com 替换为注册 Github 账号时填写的邮箱:
        ssh-keygen -t rsa -b 4096 -C "your email@example.com"
    ③ 连续敲击3次回车，即可在C:\Users\用户名文件夹\.ssh 目录中生成id rsa和 id rsa.pub 两个文件

7 配置ssh key
    ① 使用记事本打开 id rsa.pub 文件，复制里面的文本内容
    ② 在浏览器中登录 Github，点击头像 ->Settings ->SSH and GPG Keys ->New SSH key
    ③ 将id rsa.pub 文件中的内容，粘贴到 Key 对应的文本框中
    ④ 在 Title 文本框中任意填写一个名称，来标识这个 Key 从何而来

分支学习
1.分支的概念
    分支就是科幻电影里面的平行宇宙，当你正在电脑前努力学习Git的时候，另一个你正在另一个平行宇宙里努力学习SVN。
    如果两个平行宇宙互不干扰，那对现在的你也没啥影响。
    不过，在某个时间点，两个平行宇宙合并了，结果，你既学会了Git又学会了SVN!

2.分支在实际开发中的作用
    在进行多人协作开发的时候，为了防止互相干扰，提高协同开发的体验，建议每个开发者都基于分支进行项目功能的开发，各个功能开发完毕后合并到主分支。

3.master 主分支
    在初始化本地 Git 仓库的时候，Git 默认已经帮我们创建了一个名字叫做 master 的分支。通常我们把这个master 分支叫做主分支。（现在好像是main）
    在实际工作中，master主分支的作用是:用来保存和记录整个项目已完成的功能代码，
    因此，不允许程序员直接在 master 分支上修改代码，因为这样做的风险太高，容易导致整个项目崩溃

4.功能分支
    由于程序员不能直接在 master 分支上进行功能的开发，所以就有了功能分支的概念。
    功能分支指的是专门用来开发新功能的分支，它是临时从 master 主分支上分叉出来的，当新功能开发且测试完毕后，最终需要合并到 master 主分支上，

5 查看分支列表
    git branch      //打印所有分支列表 分支前的*表示当前所属的分支。

6 创建新分支
    git branch 分支名称  // 可以基于当前分支。创建一个新的分支，新分支的代码和当前分支完全一样。

7 切换分支
    git checkout 分支名称
    切换分支后展示的是分支上的文件状态 不同分支的文件数和内容肯定有差异的

8 分支的快速创建和切换
    git checkout -b 分支名称 // -b 表示创建一个新分支； checkout 表示切换到新分支

9 合并分支
    功能分支的代码开发测试完毕之后，可以使用如下的命令，将完成后的代码合并到 master 主分支上:
    git checkout master // 切换到 master 分支
    git merge login // 在 master 分支上运行 git merge 命令，将 login 分支的代码合并到 master 分支
    
    合并分支时的注意点:假设要把C分支的代码合并到 A分支则必须先切换到 A分支上，再运行 gitmerge 命令，来合并C分支!

10 删除分支
    当把功能分支的代码合并到 master 主分支上以后，就可以使用如下的命令，删除对应的功能分支:
    git branch -d 分支名称   // 删除分支 需要切换分支出去 不能你再当前分支 然后还有删除当前分支；另外需要分支合并后 才能删除分支，不合并而强制删除用-D
    git branch -D 分支名称 

11.遇到冲突时的分支合并
     # 假设:在把 reg 分支合并到 master 分支期间，代码发生了冲突
     git checkout master
     git merge reg
     # 打开包含冲突的文件，手动解决冲突之后，再执行如下的命令
     git add
     git commit -m"解决了分支合并冲突的问题"

    如果在两个不同的分支中，对同一个文件进行了不同的修改，Git 就没法干净的合并它们。此时，我们需要打开这些包含冲突的文件然后手动解决冲突。
    此时如果vscode打开冲突文件上面会出现4个按钮：采用当前更改|采用传入的更改|保留双方更改|比较变更（点击按钮即可完成修改）







git 远程分支操作

1.将本地分支推送到远程仓库
    如果是第一次将本地分支推送到远程仓库，需要运行如下的命令
    # -u 表示把本地分支和远程分支进行关联，只在第一次推送的时候需要带 -u 参数
    git push -u 远程仓库的别名 本地分支名称:远程分支名称

    # 实际案例:
    git push -u origin payment:pay

    #如果希望远程分支的名称和本地分支名称保持一致，可以对命令进行简化:
    git push -u origin payment

    git push // 不是第一次推送 这样就行

    推送完后github code面板 分支下拉选项会出现推送的分支列表

2 查看远程仓库中所有分支列表
    git remote show 远程仓库名字  // github里新建空白仓库时候的名字 实际不行 实际用的就是origin

3 跟踪分支
    跟踪分支指的是:从远程仓库中，把远程分支下载到本地仓库中。需要运行的命令如下:
    #从远程仓库中，把对应的远程分支下载到本地仓库，保持本地分支和远程分支名称相同
    git checkout 远程分支的名称
    # 示例:
    git checkout pay

    #从远程仓库中，把对应的远程分支下载到本地仓库，并把下载的本地分支进行重命名
    git checkout -b 本地分支名称 远程仓库名称/远程分支名称
    # 示例:
    git checkout -b payment origin/pay

4 拉取远程分支的最新代码
    #从远程仓库，拉取当前分支最新的代码，保持当前分支的代码和远程分支代码一致
    git pull  // 当前你在哪个分支 就下载哪个远程分支

    github上也能修改代码 code面板点进去一个文件 然后点击编辑按钮 编辑好了 输入提交描述 点击提交即可

5 删除远程分支
    #删除远程仓库中，指定名称的远程分支
    git push 远程仓库名称 --delete 远程分支名称
    # 示例:
    git push origin --delete pay
